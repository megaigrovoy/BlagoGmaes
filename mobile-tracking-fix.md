# Стабилизация трекинга на мобильных (Android)

Документ описывает проблему с распознаванием поз на Android (Samsung Galaxy S25) и то, какое именно решение помогло.

## Симптомы

- На **Android** точки запястий постоянно дрожали и **менялись местами** (левая ↔ правая).
- Ведро двигалось рывками и **переворачивалось** на 180°.
- На **iPad (Safari)** и **ПК** всё работало идеально.

## Главная причина и что помогло

**Делегат вычислений MediaPipe.** На Android по умолчанию использовался **GPU-делегат**, который загружает видеокадр как WebGL-текстуру. На многих Android-устройствах (особенно с GPU Adreno в Samsung) эта загрузка идёт с **неучтённым преобразованием ориентации кадра**. Модель получала «кривой» вход и нестабильно определяла стороны тела → точки рук дрожали и путались местами.

### Решение (то, что реально починило)

На мобильных устройствах принудительно используется **CPU-делегат** (GPU — только запасной вариант). На ПК/iPad по-прежнему GPU (там он стабилен и быстрее).

```js
// createPoseLandmarkerInstance()
const delegateOrder = trackTuning.mobile ? ['CPU', 'GPU'] : ['GPU', 'CPU'];
```

CPU-путь читает пиксели детерминированно и не путает ориентацию изображения. Именно поэтому iPad (Safari, своя реализация) и ПК были стабильны, а Android — нет.

Коммит: `ecc9ffd` — *prefer CPU pose delegate on mobile to stop Android landmark swapping*.

## Дополнительные улучшения (помогли частично / убрали другие артефакты)

Эти изменения сами по себе **не устранили** перестановку точек, но улучшили общую стабильность и убрали смежные проблемы:

1. **Сглаживание только на новом кадре камеры.**
   На экранах 120 Гц `requestAnimationFrame` вызывается 120 раз/сек, а камера выдаёт ~30 кадров/сек. Раньше сглаживание применялось на каждой отрисовке и «прокручивалось» лишние разы → дрожь. Теперь позиция ведра пересчитывается только при поступлении нового кадра камеры.
   Коммит: `6d44056`.

2. **Стабильная ориентация ведра через непрерывность во времени.**
   Вектор между запястьями не направленный, поэтому угол и угол+π — одна и та же линия. Выбирается вариант, ближайший к уже установленной ориентации, — ведро не переворачивается на 180° при перестановке запястий.
   Коммит: `6d44056`.

3. **Удержание трекинга при просадках уверенности (mobile).**
   - Порог видимости запястий `0.45 → 0.3`
   - `minTrackingConfidence`, `minPosePresenceConfidence` `0.5 → 0.3`
   - Grace-удержание ведра `280 → 600 мс`
   Коммит: `1326435`.

4. **Grace-период против мерцания.**
   При кратковременной потере трекинга ведро удерживается на последней позиции, а не исчезает и появляется снова.
   Коммит: `dc98c19`.

5. **Монотонные таймстампы для `detectForVideo`.**
   На Android `video.currentTime` иногда «прыгает» назад, из-за чего распознавание могло молча падать. Время для детекции теперь строго возрастает.

## Blasters: низкий FPS на Samsung (120 Гц экран)

### Симптом

После стабилизации трекинга (CPU-делегат) на **Samsung Galaxy S25** игра выглядела «рывковой»: перчатки, шлем, линии скелета и стрельба обновлялись ~30 раз/с, хотя экран — **120 Гц**. Враги и снаряды двигались плавно (их считает `requestAnimationFrame`), а оверлей игрока — нет.

### Причина

- `detectForVideo` и сглаживание позы выполняются **только на новом кадре камеры** (~30 fps).
- Между кадрами камеры на экране 120 Гц рисовались **те же** landmarks без интерполяции → визуально «низкий FPS».
- Дополнительное EMA на перчатках (alpha 0.15 на каждом rAF) добавляло **задержку** относительно линий трекера.

### Решение (коммит `44fb6e7`)

1. **Интерполяция позы между кадрами камеры (30 fps → 120 Hz отрисовка).**
   - На новом кадре камеры: один раз `smoothPoseLandmarks` → сохраняем `posePrevTarget` и `poseTarget`.
   - На каждом rAF: `poseDisplay = lerp(prev, target, t)` по времени с smoothstep, где `t` — доля интервала между кадрами (`poseFrameIntervalMs` из `video.currentTime`).
   - Рисуются **display**-landmarks: скелет, перчатки, жилет, шлем, стрельба, коллизии брони.

2. **Камера на Android:** `640×480`, `frameRate: { ideal: 30, max: 30 }` — быстрее CPU-инференс, меньше подвисаний game loop.

3. **Шлем на Android:** `HELMET_EAR_NOSE_SMOOTH_ALPHA` **0.38** (вместо 0.14), обновление каждый rAF по display-landmarks.

4. **Перчатки** (коммит `f1a5874`, до интерполяции): убрано лишнее EMA позиции кулака; кисти сглаживаются быстрее (`POSE_HAND_ALPHA` 0.58 на Android).

```js
// gameLoop: на новом кадре камеры
commitPoseTargetsFromFrame(orderedPersons, nowPoseMs);

// каждый rAF
updatePoseDisplayLandmarks(activePoseKeys, nowPoseMs);
// draw / fire / collision используют poseDisplayLmByPoseKey
```

### Связанные коммиты в Blasters

| Коммит | Описание |
|--------|----------|
| `358929a` | CPU-делегат MediaPipe на Android, монотонные timestamps, сглаживание по кадру камеры |
| `f1a5874` | Перчатки без лишнего EMA, быстрее следуют линиям трекера |
| `44fb6e7` | Интерполяция позы 30→120 Гц, камера 640×480, плавнее шлем и перчатки |

## Что НЕ помогло

- **Подача в MediaPipe копии кадра через 2D-канвас** (вместо `<video>`). Гипотеза была, что 2D `drawImage` исправит ориентацию для GPU-загрузки, но эффекта не дало — проблема была в самом GPU-делегате, а не в источнике кадра. Изменение откатили.
- **Понижение разрешения камеры / усиление сглаживания** на ранних итерациях только делало ведро «тормозным», но не решало перестановку точек.

## Итог

Ключ к решению нестабильного трекинга — **переключение MediaPipe на CPU-делегат на мобильных устройствах**. Остальные правки полезны для плавности и устойчивости, но корень Android-проблемы был именно в GPU-делегате и ориентации кадра.

Для **плавной картинки на 120 Гц экранах** отдельно нужна **интерполяция display-landmarks между кадрами камеры** — распознавание остаётся ~30 fps, отрисовка игрока идёт на частоте экрана.
